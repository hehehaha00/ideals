# 脑洞实验室三阶段任务清单

本文档用于指导产品经理、开发、测试按阶段执行。每个任务都可以逐项勾选，第一阶段以 MVP 为目标，先验证“AI 能否帮助用户重新进入发散状态”。

## 总体阶段划分

| 阶段 | 目标 | 核心交付 |
| --- | --- | --- |
| 第一阶段：MVP | 做出一个能玩的发散原型 | 主题输入、维度词生成、锁定/重掷、脑洞碰撞、变形、收藏 |
| 第二阶段：创意工作台 | 把单次发散变成可持续创作流程 | 素材池、联想路径、角色圆桌、孵化箱、历史项目 |
| 第三阶段：产品化 | 让它适合长期使用和对外发布 | 用户体系、模板库、分享、团队工作坊、质量评估和商业化基础 |

## 执行原则

- 第一阶段不做账号、不做团队、不做复杂商业化。
- 发散模式优先，不要一开始就做市场评分和竞品分析。
- 每个功能都要服务“让用户更容易继续想”。
- 产品经理先定义体验和文案，开发再实现，测试最后按真实使用路径验收。
- 每个阶段结束时必须产出一份阶段复盘：哪些体验有效，哪些让用户停住了。

# 第一阶段：MVP

## 阶段目标

做出一个最小可用的“AI 脑洞实验室”：用户输入一个主题后，系统能生成多组心理维度词，用户可以锁定或重掷这些词，再碰撞出脑洞卡片，并对脑洞进行变形和收藏。

## 阶段边界

第一阶段只做单人使用，不做登录，不做云同步，不做复杂项目管理，不做真实市场调研。

## 阶段验收标准

- 用户能在 10 分钟内从一个模糊主题得到至少 5 张脑洞卡片。
- 用户能看懂每张脑洞由哪些词碰撞而来。
- 用户能锁定喜欢的词，并重掷不喜欢的词。
- 用户能对脑洞执行至少 4 种变形。
- 用户能收藏脑洞，并在本地看到收藏列表。
- 测试人员能完成一条完整路径：输入主题 -> 生成词组 -> 锁词 -> 碰撞 -> 变形 -> 收藏。

## 产品经理任务

- [ ] **PM-1：定义 MVP 用户故事**
  - 输出：`MVP 用户故事清单`
  - 内容：
    - 作为灵感枯竭的用户，我可以输入一个主题，让系统帮我拆出发散词。
    - 作为用户，我可以锁定喜欢的词，避免下一轮被替换。
    - 作为用户，我可以重掷不喜欢的词，获得新的刺激。
    - 作为用户，我可以把词组碰撞成脑洞卡片。
    - 作为用户，我可以把脑洞变得更实用、更荒诞、更游戏化或更像工具。
    - 作为用户，我可以收藏脑洞，稍后继续看。
  - 验收：每条用户故事都能对应到一个页面操作。

- [ ] **PM-2：定义第一版页面结构**
  - 输出：`MVP 页面说明`
  - 页面：
    - 首页输入区：输入主题和选择发散强度。
    - 维度词区：展示人群、场景、情绪、物件、结构、限制六组词。
    - 碰撞区：展示当前被选中的词组。
    - 脑洞卡片区：展示 AI 生成的脑洞。
    - 收藏区：展示已收藏脑洞。
  - 验收：开发能根据页面说明直接拆组件。

- [ ] **PM-3：定义维度词分类**
  - 输出：`维度词分类表`
  - 第一版固定六类：
    - 人群：谁会使用。
    - 场景：什么时候使用。
    - 情绪：用户当时的心理状态。
    - 物件：和想法有关的载体或数据。
    - 结构：借来的隐喻结构。
    - 限制：故意加入的反常识条件。
  - 验收：每类至少准备 20 个示例词，方便开发做无模型兜底。

- [ ] **PM-4：定义脑洞卡片字段**
  - 输出：`脑洞卡片字段说明`
  - 字段：
    - 标题。
    - 一句话解释。
    - 碰撞来源词。
    - 为什么有趣。
    - 可以怎么变形。
    - 可能的第一版形态。
  - 验收：每张卡片不用看上下文也能独立理解。

- [ ] **PM-5：定义变形按钮**
  - 输出：`变形按钮文案和效果`
  - 第一版按钮：
    - 更实用一点。
    - 更荒诞一点。
    - 更游戏化一点。
    - 更像浏览器插件。
    - 更像 Agent skill。
    - 只保留核心隐喻。
  - 验收：每个按钮都有明确的输出变化方向。

- [ ] **PM-6：定义空状态和错误状态文案**
  - 输出：`MVP 状态文案`
  - 场景：
    - 用户还没输入主题。
    - AI 生成失败。
    - 维度词为空。
    - 收藏列表为空。
    - 输入内容太短。
  - 验收：任何空白区域都有解释，不让用户困惑。

## 开发任务

- [ ] **DEV-1：初始化前端项目**
  - 目标：搭建可运行的单页应用。
  - 建议技术：Vite + React + TypeScript + Tailwind CSS。
  - 交付：
    - 应用能本地启动。
    - 首页能显示基础布局。
    - 严格模式下无 TypeScript 错误。
  - 验收：开发环境启动后能看到“脑洞实验室”首页。

- [ ] **DEV-2：实现主题输入区**
  - 目标：用户可以输入主题并触发发散。
  - 交付：
    - 文本输入框。
    - 发散强度选择：轻微、正常、狂野。
    - 生成按钮。
    - 输入为空时禁用生成按钮。
  - 验收：用户输入“我想做一个有趣的开发者工具”后，可以点击生成。

- [ ] **DEV-3：实现维度词数据结构**
  - 目标：让六类词能被统一管理。
  - 数据结构：
    - 词文本。
    - 所属分类。
    - 是否锁定。
    - 是否被选中。
    - 来源说明。
  - 交付：
    - 支持展示词。
    - 支持锁定词。
    - 支持选择词。
    - 支持重掷未锁定词。
  - 验收：锁定的词在重掷后仍保留。

- [ ] **DEV-4：实现维度词生成**
  - 目标：根据主题生成六组词。
  - 第一版策略：
    - 优先调用 AI 生成。
    - AI 失败时使用本地示例词兜底。
    - 每类默认生成 8 个词。
  - 交付：
    - 生成中状态。
    - 生成失败提示。
    - 兜底词展示。
  - 验收：断网或接口失败时，用户仍能看到可玩的词组。

- [ ] **DEV-5：实现碰撞区**
  - 目标：用户能看到当前组合。
  - 交付：
    - 每类最多选中 1 个词。
    - 自动显示当前组合句。
    - 支持一键随机选择一组组合。
  - 验收：用户可以明确知道当前脑洞由哪些词构成。

- [ ] **DEV-6：实现脑洞卡片生成**
  - 目标：把当前组合生成 3 到 5 张脑洞卡片。
  - 交付：
    - 卡片标题。
    - 一句话解释。
    - 碰撞来源词。
    - 为什么有趣。
    - 可能的第一版形态。
  - 验收：每张卡片都能对应当前选择的词组。

- [ ] **DEV-7：实现脑洞变形**
  - 目标：对单张卡片执行不同方向的再生成。
  - 交付：
    - 变形按钮。
    - 变形中的 loading 状态。
    - 变形结果保留原始卡片来源。
  - 验收：点击“更游戏化一点”后，结果明显更接近游戏或互动玩法。

- [ ] **DEV-8：实现收藏功能**
  - 目标：用户可以保存喜欢的脑洞。
  - 第一版存储：localStorage。
  - 交付：
    - 收藏按钮。
    - 取消收藏。
    - 收藏列表。
    - 页面刷新后收藏不丢失。
  - 验收：收藏 3 张卡片并刷新页面后，收藏仍然存在。

- [ ] **DEV-9：实现基础可用性细节**
  - 目标：让 MVP 不像半成品。
  - 交付：
    - 加载状态。
    - 错误提示。
    - 空状态。
    - 基础响应式布局。
    - 按钮 hover 和 disabled 状态。
  - 验收：没有明显空白、误点和无反馈操作。

- [ ] **DEV-10：实现本地调试数据**
  - 目标：不依赖 AI 接口也能演示完整流程。
  - 交付：
    - 内置 3 个示例主题。
    - 每个主题有示例维度词。
    - 每个主题有示例脑洞卡片。
  - 验收：关闭 AI 接口后仍能走完整演示。

## 测试任务

- [ ] **QA-1：验证主流程**
  - 路径：输入主题 -> 生成维度词 -> 选择词 -> 生成脑洞 -> 变形 -> 收藏。
  - 验收：完整路径可以一次走通，无阻塞错误。

- [ ] **QA-2：验证锁定和重掷**
  - 路径：生成词 -> 锁定一个词 -> 重掷 -> 检查锁定词。
  - 验收：锁定词不变，未锁定词发生变化。

- [ ] **QA-3：验证 AI 失败兜底**
  - 路径：模拟 AI 接口失败 -> 点击生成。
  - 验收：页面显示兜底词，并给出温和提示。

- [ ] **QA-4：验证收藏持久化**
  - 路径：收藏卡片 -> 刷新页面 -> 查看收藏区。
  - 验收：收藏内容仍存在。

- [ ] **QA-5：验证空状态**
  - 场景：
    - 没输入主题。
    - 没收藏任何卡片。
    - 生成失败。
  - 验收：每个状态都有明确说明和下一步操作。

- [ ] **QA-6：验证移动端基础可用性**
  - 路径：在手机宽度下完成主流程。
  - 验收：文字不重叠，按钮可点击，核心流程可完成。

## 第一阶段完成标志

- [ ] 产品经理确认 MVP 流程闭环。
- [ ] 开发完成所有 MVP 功能。
- [ ] 测试完成主流程、异常流程、移动端基础验证。
- [ ] 产出第一阶段复盘：用户在哪一步最有继续玩的欲望，在哪一步最容易停住。

# 第二阶段：创意工作台

## 阶段目标

把第一阶段的一次性脑洞生成，升级成可持续使用的创意工作台。用户可以积累素材、查看联想路径、让不同角色点评脑洞，并把收藏的脑洞放入孵化箱继续重组。

## 阶段验收标准

- 用户能建立多个创意会话。
- 用户能把素材加入素材池。
- 用户能查看词语的联想路径。
- 用户能让角色圆桌点评同一个脑洞。
- 用户能把多个收藏脑洞混合生成新方向。

## 产品经理任务

- [ ] **PM-7：定义素材池规则**
  - 输出：`素材池设计说明`
  - 素材类型：
    - 文本。
    - 链接。
    - 手动标签。
    - 用户随手记。
  - 验收：每种素材都能被拆成维度词。

- [ ] **PM-8：定义联想路径展示方式**
  - 输出：`联想路径说明`
  - 展示内容：
    - 原始词。
    - 第一层联想。
    - 第二层联想。
    - 第三层远联想。
    - 跳转解释。
  - 验收：用户能看懂 AI 为什么从一个词跳到另一个词。

- [ ] **PM-9：定义角色圆桌**
  - 输出：`角色圆桌角色表`
  - 第一版角色：
    - 懒人。
    - 毒舌用户。
    - 产品经理。
    - 艺术家。
    - 商人。
    - 工程师。
  - 验收：每个角色都有固定关注点和固定输出格式。

- [ ] **PM-10：定义孵化箱机制**
  - 输出：`孵化箱规则`
  - 规则：
    - 用户选择 2 到 5 个收藏脑洞。
    - AI 提取共同母题。
    - AI 生成混合方向。
    - 用户可以保存为新脑洞。
  - 验收：孵化结果能追溯到原始收藏。

- [ ] **PM-11：定义会话管理**
  - 输出：`创意会话说明`
  - 内容：
    - 会话名称。
    - 原始主题。
    - 已生成词组。
    - 已生成脑洞。
    - 已收藏脑洞。
  - 验收：用户可以从历史会话继续发散。

## 开发任务

- [ ] **DEV-11：实现会话列表**
  - 目标：支持多个创意会话。
  - 交付：
    - 新建会话。
    - 切换会话。
    - 重命名会话。
    - 删除会话二次确认。
  - 验收：切换会话后，词组和收藏内容正确变化。

- [ ] **DEV-12：实现素材池**
  - 目标：用户可以持续添加素材。
  - 交付：
    - 添加文本素材。
    - 添加链接素材。
    - 给素材打标签。
    - 从素材生成维度词。
  - 验收：用户添加一段文本后，可以从中抽取新词。

- [ ] **DEV-13：实现联想路径**
  - 目标：展示 AI 的思维跳跃过程。
  - 交付：
    - 三层联想展示。
    - 每条联想的解释。
    - 从联想词加入当前词组。
  - 验收：用户可以把远联想词加入碰撞台。

- [ ] **DEV-14：实现角色圆桌**
  - 目标：让多个角色点评同一个脑洞。
  - 交付：
    - 选择角色。
    - 生成点评。
    - 保存点评。
    - 根据点评生成变体。
  - 验收：同一脑洞在不同角色下有明显不同的反馈。

- [ ] **DEV-15：实现孵化箱**
  - 目标：混合多个收藏脑洞。
  - 交付：
    - 多选收藏脑洞。
    - 生成共同母题。
    - 生成混合脑洞。
    - 保存混合结果。
  - 验收：混合结果能展示来源脑洞。

- [ ] **DEV-16：升级本地存储结构**
  - 目标：支持会话、素材、脑洞、点评和孵化结果。
  - 交付：
    - 数据版本号。
    - 旧数据迁移。
    - 导入导出 JSON。
  - 验收：第一阶段收藏数据升级后不丢失。

## 测试任务

- [ ] **QA-7：验证会话管理**
  - 路径：新建两个会话 -> 分别生成内容 -> 切换检查。
  - 验收：不同会话之间数据不串。

- [ ] **QA-8：验证素材池抽词**
  - 路径：添加一段素材 -> 抽取维度词 -> 加入碰撞台。
  - 验收：抽出的词和素材有明显关系。

- [ ] **QA-9：验证联想路径**
  - 路径：选择一个词 -> 查看三层联想 -> 加入远联想。
  - 验收：远联想能被用于新脑洞生成。

- [ ] **QA-10：验证角色圆桌**
  - 路径：选择脑洞 -> 让三个角色点评。
  - 验收：角色输出不重复，关注点不同。

- [ ] **QA-11：验证孵化箱**
  - 路径：选择多个收藏 -> 生成混合方向 -> 保存。
  - 验收：新方向能看到来源脑洞。

- [ ] **QA-12：验证数据迁移**
  - 路径：准备第一阶段旧数据 -> 打开第二阶段版本。
  - 验收：旧收藏可以正常显示。

## 第二阶段完成标志

- [ ] 产品经理确认创意工作台闭环。
- [ ] 开发完成会话、素材池、联想路径、角色圆桌、孵化箱。
- [ ] 测试完成历史数据、异常流程和多会话验证。
- [ ] 产出第二阶段复盘：哪些机制最能促进用户继续发散。

# 第三阶段：产品化

## 阶段目标

把脑洞实验室从个人原型升级为可以长期使用、可以分享、可以对外发布的产品。

## 阶段验收标准

- 用户可以登录并同步数据。
- 用户可以使用模板快速开始。
- 用户可以分享脑洞卡片或会话。
- 团队可以进行一次远程脑洞工作坊。
- 产品能记录基本使用数据，用于判断哪些发散机制有效。

## 产品经理任务

- [ ] **PM-12：定义用户体系和数据归属**
  - 输出：`账号与数据规则`
  - 内容：
    - 游客模式。
    - 登录用户。
    - 私有会话。
    - 可分享会话。
  - 验收：用户知道哪些内容只属于自己，哪些可以分享。

- [ ] **PM-13：定义模板库**
  - 输出：`模板库分类`
  - 模板方向：
    - 项目灵感。
    - 游戏创意。
    - 内容选题。
    - 产品概念。
    - 设计方向。
    - Agent skill 脑洞。
  - 验收：用户可以不输入主题，直接从模板开始。

- [ ] **PM-14：定义分享机制**
  - 输出：`分享规则`
  - 分享对象：
    - 单张脑洞卡。
    - 一组碰撞词。
    - 一个完整会话。
  - 验收：分享内容不暴露用户私有素材。

- [ ] **PM-15：定义团队工作坊模式**
  - 输出：`工作坊流程`
  - 流程：
    - 主持人创建主题。
    - 参与者各自投词。
    - 系统生成公共碰撞池。
    - 大家投票收藏脑洞。
    - 输出工作坊报告。
  - 验收：3 到 6 人能完成一次 30 分钟线上发散。

- [ ] **PM-16：定义质量评估指标**
  - 输出：`脑洞质量指标`
  - 指标：
    - 用户收藏率。
    - 用户变形次数。
    - 用户回访率。
    - 单次会话生成脑洞数量。
    - 用户手动编辑比例。
  - 验收：指标能帮助判断产品是不是让用户继续想。

## 开发任务

# 跨阶段总验收清单

- [ ] 用户能从“我不知道做什么”开始，而不是必须先有明确需求。
- [ ] 产品能展示 AI 的联想过程，而不是只给结论。
- [ ] 用户能参与选择、锁定、重掷、变形和收藏。
- [ ] 产品允许荒诞脑洞存在，不强行变成严肃商业计划。
- [ ] 收藏和孵化能让灵感跨会话延续。
- [ ] 第一阶段像玩具，第二阶段像工作台，第三阶段像产品。
