# LLM 编辑部圆桌讨论设计

## 目标

在单篇脑洞报告中增加一个按需开启的多角色讨论环节，让不同的思维角色围绕同一个脑洞制造有差异的观点、冲突和新火花。讨论结束后，用户可以把火花采集回无限画布，继续发散。

## 产品边界

- 默认关键词画布不展示讨论入口。
- “换个立场”继续作为单角色、快速反共识检查。
- “召集讨论”是报告页的深度次级动作，不替代“深入验证”。
- 第一版固定四个思维角色：用户代言人、反常识派、跨界连接者、现实构建者；主持人由系统隐藏执行。
- 讨论固定三轮：各自判断、冲突回应、方向收束。
- 每个角色每轮最多一条短观点，每条观点最多一个可采集火花。
- 讨论不会自动把所有内容写回画布，只有用户主动采集才生成节点。
- 桌面端优先，手机端不纳入本轮验收。

## 角色与产出

角色不是聊天人格，而是稳定的思维动作：

- 用户代言人：指出真实使用动机、首次使用摩擦和触发场景。
- 反常识派：拆掉默认前提，提出仍然自洽的反转方向。
- 跨界连接者：从其他产品、行业、游戏、仪式或物件借用结构。
- 现实构建者：指出最小可行版本、失败路径和可验证的第一步。

每个观点包含：`role`、`claim`、`tension`、可选 `spark`、可选 `buildsOn`。主持人最终输出：`conservativeDirection`、`radicalDirection`、`unexpectedDirection`，每个方向包含标题、说明和下一步。

## 讨论流程

1. 用户在脑洞报告点击“召集讨论”。
2. 系统保存当前脑洞快照，进入 `discussion` loading 状态并锁定报告操作。
3. 单次结构化模型请求模拟完整三轮：第一轮四个角色分别判断，第二轮围绕主要张力回应，第三轮由主持人收束方向。
4. 前端收到完整结果后按轮次呈现，并标记可采集火花。
6. 用户可以采集任意火花、继续查看讨论或结束讨论。
7. 采集火花时在当前脑图中新增一个普通可选节点；没有可用脑图时只保存在讨论记录中并提示稍后返回画布。

## UI 结构

- 报告页保留现有来源星座、挑战和深入验证；“召集讨论”放在挑战区域之后，作为次级入口。
- 讨论面板使用编辑部批注的连续版式，不使用聊天气泡堆叠。
- 顶部显示脑洞标题、当前轮次和停止按钮。
- 每条观点显示角色标签、判断、冲突点；火花使用轻量的“采集”按钮。
- 底部显示三个收束方向，点击方向只选中，不自动生成执行计划。
- AI 工作期间禁用角色选择、变形、收藏、深入验证和采集按钮；取消后保留已收到的讨论内容，不伪造收束结果。

## 数据与持久化

新增 `IdeaDiscussion`：

- `id`、`ideaId`、`createdAt`、`status`
- `participants`、`rounds`、`synthesis`
- `collectedSparkIds`

工作区新增 `discussionsByIdeaId: Record<string, IdeaDiscussion[]>`，允许同一脑洞保留多次讨论；孵化条目同步保存对应讨论。旧 V2 工作区没有此字段时恢复为空对象；未知角色、轮次或火花字段单独过滤，不影响脑洞、炼化和挑战结果。

## API 与可靠性

- 新增 `POST /api/idea/discussion`，请求包含当前脑洞和固定讨论配置。
- 服务端严格校验角色、轮次、观点和收束方向；缺失结构返回明确错误。
- 服务端沿用现有 SSE 文本进度和最终结构化结果，不拆分为多次模型调用，避免成本、排队和过时响应成倍增加。
- 前端沿用现有请求编号、状态修订号、AbortController 和队列截止时间，过时响应不能覆盖新讨论或其他报告操作。
- 讨论失败时保留已收到的观点和旧讨论记录，不写入不完整的 synthesis。

## 非目标

- 不做无限自由聊天。
- 不允许用户在第一版创建几十个角色。
- 不做语音会议、多人协作和跨脑洞同时讨论。
- 不自动把所有讨论文本转为画布节点。

## 验收标准

- 默认关键词画布不出现讨论入口。
- 报告中点击“召集讨论”后能看到三轮状态和角色观点。
- 至少一条观点可采集，采集后产生一个新画布节点并保留讨论记录。
- 讨论期间所有报告操作锁定，取消和失败不会覆盖已有脑洞、挑战或炼化结果。
- 刷新后讨论记录和已采集火花状态仍在。
- API、Store、存储、组件和 Chromium 桌面主流程均有测试覆盖。
