## 语言与风格
用简洁、非技术的语言描述完成的内容，让非技术背景的人也能看懂。任务完成后给一个简短的总结，说清楚做了什么、结果是什么。中间过程如果要说，也用同样的风格，不需要列出每个技术细节。测试结果只报告通过/失败数量，不逐条列出用例。用中文回复我。

## 自主度
明确的子任务可以自主完成，不需要每步确认。遇到真正需要决策的分叉点再停下来问我。不要在任务完成后主动提出优化建议或替代方案。如果有重要问题需要注意，最多一句话注明，不展开。

## 文件处理
所有文件读写使用 UTF-8 编码，修改文件时不要改变原有编码。在 PowerShell 中读取含中文的文件时，先执行 `chcp 65001` 并设置 UTF-8 输出，读取时用 `Get-Content -Encoding UTF8`。不要用 sed/awk 处理含中文的文件，改用 Python 或 Node.js。代码注释使用中文。

## Python 环境
在项目中运行 Python 时，优先使用项目本地的虚拟环境。Windows 用 `.venv/Scripts/python.exe`，Unix 用 `.venv/bin/python`，不要直接用 `python` 命令。如果 `.venv` 不存在，先用 `python -m venv .venv` 创建再运行。

## 项目开始前
每次开始一个新项目或新会话时：
1. 先读 CONTEXT.md 了解当前进度和关键决定（如有），这是最轻量的启动方式
2. 根据任务需要再决定是否读 README.md 和 ARCHITECTURE.md，不需要每次都全读
3. 检查项目根目录是否有 AGENTS.md，有则优先读取其中的项目规则
4. 查看 README.md 里是否已有搜索记录，有则跳过搜索；没有且是新功能或新项目，才访问 https://skills.sh/ 和 GitHub 搜索可参考方案，并把搜索结论更新到 README.md 的搜索记录章节
5. 简单任务读完 CONTEXT.md 即可直接开始；复杂任务或新功能才需要读完整文档

## 前端通用编码规则与技术选型

以下规则适用于现代企业级前端项目，强调类型安全、分层清晰、交互一致、性能可靠和可维护性。

### TypeScript 与 API 规范（强制）
- TypeScript 必须开启严格模式，禁止使用 `any`；不确定类型先用 `unknown`，使用前做类型收窄。
- 所有变量、函数参数、返回值、`useState`、事件回调都必须有明确类型。
- API 请求和响应必须定义明确的 `interface` 或 `type`，确保前后端字段对齐。
- 所有后端 API 调用必须通过统一的 `services/api.ts` 或对应业务 service，不要在页面或组件里直接写 `fetch()` / `axios`。
- 新增接口先在 service 层定义请求/响应类型和方法，再在页面或 Hook 中使用。
- API 层统一处理 token 注入、错误处理、超时、响应格式、AbortController 和 401 刷新逻辑。

### 组件、Hook 与目录规范
- 目录结构优先按功能模块和职责分层组织，推荐保留 `services` / `hooks` / `components` / `views` / `types` 等边界。
- 页面组件负责数据获取、状态组合和业务编排；通用组件只通过 `props` 接收数据，并通过回调向上抛出事件。
- 可复用的数据获取、状态管理、交互逻辑必须下沉到自定义 Hook，避免页面重复写 loading/error/请求逻辑。
- React Hooks 必须遵守依赖数组规则，列表渲染必须使用稳定 `key`。
- 复杂对象或函数向下传递时，按需使用 `useMemo` / `useCallback`，避免无意义重渲染，但不要滥用。
- mock 数据仅用于开发初期的 UI 原型，正式功能必须接入真实 API。

### 样式、交互与无障碍规范
- Tailwind 优先，避免行内样式；只有动态计算属性才允许使用 `style`。
- 条件类名统一使用 `cn()`（clsx + tailwind-merge），避免模板字符串导致样式冲突。
- 可点击元素必须有 hover / active 状态；重要操作要有明确反馈。
- 加载状态优先使用 Skeleton，空状态必须有清晰说明和引导操作。
- 删除、作废、批量修改等破坏性操作必须二次确认。
- 表单提交失败必须给出字段级错误提示。
- 表单、弹窗、下拉菜单等交互组件必须支持键盘操作：Tab 切换、Enter/Space 选中、Esc 关闭。

### 推荐技术选型
- 构建工具优先使用 Vite。
- UI 方案优先使用 shadcn/ui + Radix UI + Tailwind CSS。
- 图标优先使用 Lucide React，保持风格一致和体积轻量。
- 服务端状态管理优先使用 TanStack Query，用于缓存、分页、后台刷新、预取和乐观更新。
- 客户端 UI 状态优先使用 Zustand，例如侧边栏、主题、多语言、筛选器草稿等。
- 表单优先使用 React Hook Form + Zod，实现高性能表单和统一校验。
- 路由优先使用 TanStack Router 或 React Router v6；需要类型安全时优先 TanStack Router。
- 大列表、审计日志、知识列表、任务列表使用虚拟列表，优先考虑 `@tanstack/react-virtual`。
- 路由级页面必须支持懒加载，使用 `React.lazy` 和 `Suspense` 降低首屏体积。
- 搜索输入、窗口 resize、实时筛选等高频操作必须使用防抖或节流。
- 审核任务、知识状态流转、冲突处理等流程型页面可使用看板交互，拖拽库优先考虑 `@hello-pangea/dnd`。
- 富文本场景可考虑 TipTap 或 Quill，但仅在确实需要编辑富文本时引入。

### 本项目优先落地的前端能力
- RBAC 权限控制和按权限渲染菜单、按钮、操作项。
- 高级筛选器支持知识健康仪表盘，例如筛选过期、高风险、无 owner、待审核知识。
- Review Center 和知识治理任务可采用看板式状态流转。
- 虚拟列表处理大量知识、任务和审计记录。
- Playwright 覆盖登录、权限、知识审核、问答引用等关键流程。

## 后端 Python 规范
- 所有 Service 方法必须有完整的异常处理，不允许裸 `except` 或静默吞错
- 路由层（routes/）只做参数校验和响应格式化，业务逻辑必须放在 Service 层
- 数据库操作必须通过 ORM，禁止在路由中直接写 SQL
- Celery 任务必须有重试机制和失败回调
- 临时调试脚本不得以 `test_` 开头（会被 pytest 误执行），应放在 `scripts/` 目录或使用其他命名

## 测试规范
- 新增的 Service 必须附带单元测试
- 新增的路由必须附带接口测试
- 测试文件放在 `tests/` 对应子目录下，命名 `test_<模块名>.py`
- 测试用例必须能独立运行，不依赖外部服务（ASR/LLM 等用 mock）
